## Scope

This document defines the official Architecture Decision Record process for
the project.

It explains when to create an ADR, how to name it, how to write it, how to
validate it, and how to replace or deprecate an architecture decision.

No real ADR is created by this document.

## Purpose

An ADR records a durable architecture decision.

It should explain:

- The context.
- The options that were considered.
- The selected decision.
- The consequences.
- The security impact.
- The performance impact.

An ADR mainly explains why a decision exists.

Thematic documentation mainly describes what exists and how the system works.

## What is an ADR?

An ADR is a short document that keeps the reasoning behind an architecture
choice visible over time.

It should be used when future developers, reviewers, or assistants need to know
why the project chose one direction instead of another.

## When to create an ADR?

Create an ADR when a decision affects, for example:

- Gameplay authority source.
- Client/server boundaries.
- Network or Socket.IO architecture.
- Chunk format or chunk size.
- Mobility or collision strategy.
- Persistence strategy.
- Important data model.
- Major dependency.
- Stack change.
- Security.
- Authentication or authorization.
- Admin architecture.
- Asset or map strategy with durable impact.
- A choice with important maintenance cost.
- Replacement of an existing decision.

Create an ADR when the reasoning matters beyond the immediate code change.

## When not to create an ADR?

Do not create an ADR for:

- Local bug fixes.
- Text changes.
- Formatting.
- Minor renames.
- Small refactors without architecture change.
- Reversible implementation details.
- Experiments that were not retained.
- Descriptive documentation of existing behavior.

## ADR naming convention

Use this format:

```text
ADR-NNN-short-kebab-case-title.md
```

Illustrative examples:

```text
ADR-001-server-authoritative-gameplay.md
ADR-002-world-chunk-size.md
ADR-003-client-server-movement-validation.md
```

Rules:

- Use increasing numbers.
- Never reuse a number.
- Keep the title short.
- Use technical English.
- Keep the file even if the decision is later replaced.

Do not create these example files unless the matching decisions are actually
being recorded.

## ADR lifecycle

An ADR has two distinct lifecycle fields:

- `Status`: general documentation lifecycle, such as `Draft`, `Review`, or
  `Stable`.
- `Decision status`: decision lifecycle, such as `Proposed`, `Accepted`,
  `Superseded`, `Deprecated`, or `Rejected`.

ADR-specific statuses:

- `Proposed`: decision is written but not validated.
- `Accepted`: decision is validated and applicable.
- `Superseded`: decision was replaced by another ADR.
- `Deprecated`: decision should no longer be used.
- `Rejected`: proposal was considered but not selected.

The general documentation status can still follow the project documentation
lifecycle, but each ADR must also have its own `Decision status` field.

- `Status: Stable` means the ADR document was reviewed and validated as a
  document.
- `Decision status: Accepted` means the decision was approved.
- An `Accepted` decision does not prove that its implementation is complete.
- The real implementation state must always be checked in the code and in
  project status documents.
- No assistant can modify either status alone toward a validated state.

An assistant cannot accept, reject, supersede, or deprecate an ADR alone.

## Validation process

Use this process:

```text
1. Identify the decision need.
2. Analyze the existing project.
3. Check previous ADRs.
4. Write the context and options.
5. Evaluate security, performance, and maintenance.
6. Request review.
7. Obtain explicit human validation.
8. Set Decision status to Accepted.
9. Update related documents.
10. Implement or align code according to the validated decision.
```

An ADR MUST NOT be marked `Accepted` only because it was generated by an
assistant.

Acceptance authorizes implementation or code alignment. It is not proof that
the implementation is complete.

Descriptive documentation and `STATUS.md` must separately reflect the real
implementation state. Any divergence between an accepted ADR and the code must
be reported.

The template must stay generic. It must not contain a real project decision
until an ADR is created for a validated need.

`Approved by` must identify the human owner or role that validated the
decision.

`Approval reference` can point to a commit, issue, review, or other durable
trace.

Do not record sensitive personal data.

`Date accepted` remains `N/A` while the decision is `Proposed`.

## Replacing an ADR

When a decision changes:

- Do not delete the old ADR.
- Create a new ADR.
- Set the old ADR's `Decision status` to `Superseded`.
- Set `Superseded by` in the old ADR.
- Set `Supersedes` in the new ADR.
- Keep the old ADR file in the same folder.
- Keep its number and filename unchanged.
- Preserve its Git history.
- Reference the old ADR precisely from the new ADR.
- Update related documents.
- Explain migration and compatibility consequences.
- Do not present the old ADR as active architecture anymore.

The old ADR remains useful as historical context.

## Rejected decisions

A `Rejected` ADR can be kept when:

- The analysis has historical value.
- The option may be proposed again later.
- The rejection reasons should remain visible.

A `Rejected` ADR must not be presented as active architecture.

## Security requirements

An ADR that touches Phaser client, mobility, collisions, maps, Tiled, Zustand,
Socket.IO, admin, permissions, resources, loot, or inventories must explicitly
evaluate the trust model.

When gameplay is involved, the ADR must confirm that the server remains
authoritative.

These requirements do not claim that any validation is already implemented.

## Performance requirements

An ADR should examine relevant impacts:

- Server load.
- Network traffic.
- Memory.
- SQL queries.
- Concurrency.
- Chunk volume.
- Entity volume.
- Processing frequency.
- MMORPG scalability.

Use `N/A` with a short justification when a point is not relevant.

## Non-goals

- This document does not create any decision.
- This document does not replace architecture documentation.
- This document does not replace tests.
- This document does not guarantee that a decision is correctly implemented.
- This document does not give assistants the right to validate a decision.
- This document does not create automation tools.

## Security notes

Human validation is required for accepted decisions.

ADRs must not contain secrets, credentials, real environment values, or
sensitive data.

Security-sensitive ADRs must explicitly analyze the trust model.

An ADR must not give gameplay authority to the client.

## Performance notes

This document has no runtime impact.

ADRs should document relevant performance impacts before a durable architecture
choice is accepted.
